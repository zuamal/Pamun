# F6 — Impact Analysis

## 개요

`changed` 플래그가 설정된 요구사항을 기준으로 1-hop 영향 분석을 수행하고 결과를 반환하는 API를 구현한다.

## 완료 기준

- [ ] `GET /api/impact` — 현재 `changed=true`인 요구사항 기준 1-hop 영향 분석 반환
  - `changed=true`인 요구사항이 없으면 `affected_items=[], review_items=[], total_affected=0, total_review=0` 반환 (오류 아님)
  - `APPROVED` 상태의 Edge만 분석에 사용
  - `changed=true`인 요구사항 자체는 결과 목록에 포함하지 않음
- [ ] 방향성 정책이 정확히 구현된다:
  - `related_to` Edge: 양쪽 모두 `affected`
  - `depends_on` Edge에서 **target이 changed**: source → `affected`
  - `depends_on` Edge에서 **source가 changed**: target → `review_recommended`
- [ ] 동일 요구사항이 여러 changed 노드에서 영향받을 경우, 더 높은 impact_level(`affected` > `review_recommended`)로 단일 항목으로 반환
- [ ] 각 `ImpactItem`에 `document_filename`, `display_label`, `evidence`, `char_start`, `char_end` 포함
- [ ] `total_affected` = `affected_items` 길이, `total_review` = `review_items` 길이
- [ ] `cd backend && mypy app/ --strict` 오류 없음
- [ ] `pytest tests/test_impact.py -v` 통과:
  - changed 없음 케이스
  - `depends_on` 방향별 케이스 (target changed vs source changed)
  - `related_to` 양방향 케이스
  - PENDING Edge가 분석에 포함되지 않는 케이스
  - 중복 요구사항 dedup + impact_level 우선순위 케이스

## 구현 범위

```
backend/
├── app/
│   ├── api/
│   │   └── impact.py         # GET /api/impact
│   └── services/
│       └── impact_service.py # 영향 분석 로직 (networkx 탐색)
├── tests/
│   └── test_impact.py
```

**구현 참고:** networkx 그래프를 직접 순회하기보다 `AppStore.edges` dict를 기반으로 필터링하여 구현하는 것도 가능. `APPROVED` edge만 networkx에 추가된 상태이므로 graph traversal이 자연스럽다.
