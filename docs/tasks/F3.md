# F3 — LLM Requirement Parsing

## 개요

Anthropic Claude + Instructor를 사용해 업로드된 문서에서 요구사항을 파싱하는 LLM 파이프라인과 API를 구현한다.

## 완료 기준

- [ ] `POST /api/parse` — 지정된 document_id 목록을 파싱하여 `Requirement` 목록 반환
  - `document_ids`에 존재하지 않는 ID 포함 시 422 반환
  - `document_ids`가 비어 있으면 422 반환
  - 파싱 성공 시 스토어에 `Requirement` 저장, 응답으로 목록 반환
  - 이전 파싱 결과(동일 document_id)는 덮어쓴다 (재파싱 허용)
- [ ] `GET /api/requirements` — 전체 요구사항 목록 반환
- [ ] LLM 파싱 결과가 `ParseResult` 스키마(`InferredEdge` 아님)를 준수한다
  - 각 `ParsedRequirement`의 `char_start`/`char_end`가 `raw_text` 범위 내에 있다
  - `original_text`가 `raw_text[char_start:char_end]`와 일치한다
- [ ] `ANTHROPIC_API_KEY` 미설정 시 서버 기동 단계에서 명확한 오류 메시지 출력
- [ ] Instructor retry 횟수: 최대 3회 (스키마 위반 시 자동 재시도)
- [ ] `cd backend && mypy app/ --strict` 오류 없음
- [ ] `pytest tests/test_parse.py -v` 통과:
  - 실제 LLM 호출 없이 Instructor 응답을 mock하여 파싱 로직 검증
  - `char_start`/`char_end` 범위 검증
  - 없는 document_id 오류 케이스

## 구현 범위

```
backend/
├── app/
│   ├── api/
│   │   └── parse.py          # POST /api/parse
│   ├── llm/
│   │   ├── __init__.py
│   │   ├── schemas.py        # ParsedRequirement, ParseResult (ADR Appendix A.2)
│   │   └── parser.py         # parse_document(doc: Document) → list[Requirement]
│   └── services/
│       └── parse_service.py  # 여러 문서 파싱 오케스트레이션
├── tests/
│   └── test_parse.py
└── .env                      # ANTHROPIC_API_KEY, LLM_MODEL=claude-sonnet-4-5
```

**LLM 모델 설정:** `backend/.env`의 `LLM_MODEL` 환경변수로 주입. 기본값 `claude-sonnet-4-5`.

**Requirement ID 생성:** `uuid4()` 사용.

**파싱 프롬프트 방향:**
- 헤더가 있는 문서: 헤더 단위로 분리 우선
- 헤더 없는 문서: LLM이 의미 단위로 자율 분리
- 요구사항 수 상한: 50개 (프롬프트에 명시)
