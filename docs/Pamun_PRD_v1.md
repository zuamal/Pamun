## 1. Overview

### 1.1 Problem Statement

PM이 여러 기획 문서(PRD, 기능명세서 등)를 작성·관리할 때 두 가지 핵심 문제가 반복적으로 발생한다:

1. 요구사항 간 의존관계가 문서 여러 곳에 흩어져 있어, 특정 기능을 변경할 때 어떤 문서의 어떤 부분이 영향받는지 파악하기 어렵다.
2. 수정 시 영향 범위를 놓치면 문서 간 불일치가 발생하고, 이는 개발·QA 단계에서 비용이 큰 재작업으로 이어진다.

### 1.2 Solution

Pamun은 기획 문서 묶음을 입력받아, LLM이 요구사항 단위로 파싱하고 문서 간/내 의존관계를 추론한 뒤, PM이 특정 요구사항에 변경을 표시하면 영향받는 문서의 섹션을 자동으로 찾아주는 도구다.

### 1.3 Core Value Proposition

"변경 하나가 어디까지 영향을 미치는지"를 LLM이 자동으로 추적하고, PM을 해당 문서 위치로 데려다 주는 것. 수동으로 문서를 뒤져가며 찾는 시간을 제거한다.

### 1.4 Target User

- **역할:** PM / Product Owner
- **사용 맥락:** 문서 3–5개, 요구사항 수십 개 규모의 프로젝트
- **페인 포인트:** 기능 변경 시 관련 요구사항을 수동으로 찾아 다니며 영향 범위를 놓치는 것

### 1.5 Project Context

- **목적:** 포트폴리오용 오픈소스 프로젝트
- **우선순위:** 데모 가능성 > 높은 정확도 > 대규모 지원
- 기술적 의사결정의 근거를 문서화하여 포트폴리오로서의 가치를 높인다

## 2. User Flow

MVP의 전체 사용 흐름을 단계별로 정리한다.

| Step | 단계 | 설명 |
|------|------|------|
| 1 | 문서 업로드 | PM이 기획 문서 묶음(3–5개)을 업로드한다. 지원 파일 포맷: Markdown, Docx, PDF. 문서 구조는 자유형식. |
| 2 | 요구사항 파싱 | LLM이 문서를 분석하여 요구사항 단위로 추출한다. 각 요구사항에 ID, 요약 제목, 원문 범위(문서명 + 위치), 표시용 라벨을 부여한다. |
| 3 | 파싱 결과 검토 | PM이 추출된 요구사항 목록을 확인하고, 분리/병합/삭제/수정할 수 있다. |
| 4 | 의존관계 추론 | LLM이 요구사항 간 의존관계를 추론한다. 각 연결에 근거 문장을 표시한다. |
| 5 | 연결 검토/수정 | PM이 추론된 연결을 승인/거부/수정하고, 새 연결을 수동 추가할 수도 있다. |
| 6 | 그래프 확정 | 승인된 연결을 기반으로 의존관계 그래프가 시각화된다. |
| 7 | 변경 마킹 | PM이 그래프에서 특정 노드(요구사항)에 "변경 예정" 플래그를 설정한다. 여러 노드에 순차적으로 설정 가능. |
| 8 | 영향 분석 | 변경된 노드와 연결된 모든 요구사항이 하이라이트된다. 각 항목의 원문 문서 위치를 표시하고, 클릭 시 해당 섹션으로 이동한다. |
| 9 | 문서 수정 | PM이 영향받는 섹션을 확인하고 원문 문서를 직접 수정한다 (도구 외부). |

## 3. Functional Requirements

### 3.1 문서 업로드 및 파싱

| ID | Requirement |
|----|-------------|
| FR-1.1 | 지원 파일 포맷: Markdown, Docx, PDF. 문서 구조(헤더, 문단 등)는 자유형식을 허용한다. |
| FR-1.2 | 한 번에 최대 5개 문서 업로드. 총 요구사항 수 상한은 MVP에서 50개로 제한. |
| FR-1.3 | LLM이 문서를 요구사항 단위로 파싱하여 추출. 각 요구사항에 자동 ID, 요약 제목, 원문 위치(문서명 + char offset), 표시용 라벨(display_label)을 부여한다. |
| FR-1.4 | 파싱 기준: 헤더가 있으면 헤더 단위를 우선 적용하고, 헤더가 없으면 LLM이 의미 단위로 자율 분리한다. 파싱 결과는 PM이 최종 검토한다. |
| FR-1.5 | PM이 파싱 결과를 검토하며 요구사항을 분리, 병합, 삭제, 제목 수정할 수 있다. 병합은 같은 문서의 인접 span끼리만 허용한다. 분리는 하나의 span을 두 개로 나누는 것만 허용한다. |

### 3.2 의존관계 추론 및 검토

| ID | Requirement |
|----|-------------|
| FR-2.1 | LLM이 요구사항 간 의존관계를 추론한다. 관계 유형: depends_on, related_to. |
| FR-2.2 | 각 연결에 근거 문장(원문에서 발췌)을 함께 표시한다. |
| FR-2.3 | PM이 각 연결을 승인/거부할 수 있다. |
| FR-2.4 | PM이 LLM이 찾지 못한 연결을 수동으로 추가할 수 있다. 수동 추가된 연결은 자동으로 승인 상태가 된다. |
| FR-2.5 | 승인된 연결을 사후에 수정·삭제할 수 있다. 연결 수정 시 그래프에 즉시 반영된다. |

### 3.3 그래프 시각화

| ID | Requirement |
|----|-------------|
| FR-3.1 | 승인된 연결만으로 구성된 요구사항 의존관계 그래프를 표시한다. |
| FR-3.2 | 노드: 요구사항 ID + 요약 제목. 엣지: 관계 유형(depends_on / related_to). 문서별 클러스터링으로 시각적 구분. |
| FR-3.3 | 노드 클릭 시 해당 요구사항 상세 패널 표시: 원문 범위, 연결된 요구사항 목록, 근거 문장. |
| FR-3.4 | 그래프에서 직접 연결을 추가/삭제할 수 있다 (FR-2.4, FR-2.5의 UI). |

### 3.4 변경 영향 분석 (Impact Analysis)

| ID | Requirement |
|----|-------------|
| FR-4.1 | PM이 그래프에서 특정 노드에 "변경 예정" 플래그를 설정한다. 의미: "이 요구사항이 변경될 예정이다." 여러 노드에 순차적으로 플래그를 설정한 뒤, 한 번에 영향 분석을 실행할 수 있다. 원문 수정은 도구 외부에서 PM이 직접 수행한다. |
| FR-4.2 | 플래그가 설정된 노드에 대해, 의존관계로 연결된 요구사항을 다음 정책에 따라 하이라이트한다: (1) `related_to` 연결: 양방향 영향 — 양쪽 모두 하이라이트. (2) `depends_on` 연결: target이 변경되면 source는 **영향받음**으로 하이라이트. source가 변경되면 target은 **검토 권장**으로 하이라이트. |
| FR-4.3 | 영향받는 각 요구사항에 대해: 원문 문서명, 표시용 위치 라벨(display_label), 근거 문장을 표시한다. |
| FR-4.4 | 영향받는 항목 클릭 시 원문 문서의 해당 섹션으로 이동(jump-to-section)한다. |
| FR-4.5 | 영향 분석은 직접 연결(1-hop)만 표시한다. N-hop 전파는 v2에서 검토한다. |

## 4. Key Design Decisions

### 4.1 요구사항 파싱 기준

- **문제:** 문서마다 작성 스타일이 달라 "요구사항 단위"의 경계가 모호하다.
- **결정:** 헤더 우선 + LLM 자율 분리 + PM 검토 3단계 접근. 헤더가 있으면 헤더 단위, 없으면 LLM이 의미 단위로 분리하고, PM이 최종 확정한다.
- **근거:** 포트폴리오용 프로젝트로서 정확도보다 데모 가능성을 우선하며, PM 검토를 통해 오류를 보정한다.

### 4.2 변경 마킹 방식

- **문제:** PM이 노드에 변경을 표시할 때, 요구사항 내용을 직접 수정하는 것인지 플래그만 설정하는 것인지 모호하다.
- **결정:** "변경 예정" 플래그 방식을 채택한다. 원문 수정은 도구 외부에서 PM이 직접 수행한다.
- **근거:** 도구 내에서 원문을 수정하면 원본 문서와의 싱크 문제가 발생하고, MVP 범위에서 양방향 동기화는 복잡도가 너무 높다. 도구의 역할을 "영향 범위 파악"에 한정하고, 수정 책임은 PM에게 둔다.

### 4.3 섹션 정의 기준

- **문제:** "영향받는 섹션"을 하이라이트한다고 했는데, 섹션의 경계가 문서마다 다르다.
- **결정:** 섹션 = 파싱 시 추출한 요구사항의 원문 범위와 동일하게 정의한다.
- **근거:** 별도의 섹션 정의 로직을 만들면 파싱 결과와 영향 분석의 단위가 달라져서 혼란이 생긴다. 파싱과 영향 분석의 단위를 통일하여 일관성을 확보한다.

### 4.4 문서 재업로드 전략 (MVP)

- **문제:** PM이 원문 수정 후 문서를 다시 업로드하면 기존 그래프와 승인된 연결은 어떻게 되나.
- **결정:** MVP에서는 재업로드 시 그래프를 초기화하고 전체 파이프라인을 다시 실행한다.
- **근거:** 문서 변경 부분만 감지하여 그래프를 부분 업데이트하는 것은 diff 알고리즘 + 요구사항 매핑 동기화가 필요하여 MVP 범위를 초과한다. v2에서 diff 기반 부분 업데이트를 검토한다.

### 4.5 분리/병합 제약

- **문제:** 요구사항을 자유롭게 분리·병합하면 원문 위치 추적이 복잡해진다.
- **결정:** MVP에서 병합은 같은 문서의 인접 span끼리만 허용하고, 분리는 하나의 span을 두 개로 나누는 것만 허용한다.
- **근거:** 복수 source span을 갖는 모델은 jump-to-section, 영향 분석, 그래프 표시 등 모든 하위 시스템을 복잡하게 만든다. MVP 단순성을 유지하면서 기본적인 편집 기능을 제공한다. 교차 문서 병합 및 복수 span 지원은 v2에서 검토한다.

## 5. Scope

### MVP (v1) — 포함

- 문서 업로드 (Markdown, Docx, PDF / 최대 5개)
- LLM 기반 요구사항 파싱 + PM 검토 (분리/병합은 인접 span 제한)
- 의존관계 추론 + 근거 문장 표시 + PM 승인/수정
- 의존관계 그래프 시각화 (문서별 클러스터링)
- 변경 예정 플래그 + 1-hop 영향 분석 (방향성 정책 적용)
- 영향받는 섹션으로 jump-to-section 이동

### MVP (v1) — 제외

- 문서 버전 관리 및 diff 기반 부분 업데이트 (v2)
- N-hop 영향 전파 (v2)
- 도구 내 문서 직접 수정 기능
- 교차 문서 병합, 복수 source span (v2)
- 실시간 협업 (동시 편집)
- 외부 툴 연동 (Jira, Confluence 등)

## 6. Risks & Mitigations

| 리스크 | 영향 | 완화 전략 |
|--------|------|-----------|
| LLM 파싱 정확도가 낮아 요구사항 경계가 부정확함 | 그 위에 쌓이는 의존관계와 영향 분석이 모두 흔들림 | PM 검토 단계를 필수로 두어 오류 보정. 헤더 기반 파싱으로 기본 정확도 확보 |
| PM이 검토해야 할 연결이 많아 검증 비용이 높음 | "LLM이 자동화해준 시간보다 검증에 더 많은 시간 소요" 역설 | MVP에서 요구사항 50개 상한으로 제한. 신뢰도 점수 기반 정렬로 검토 우선순위 제공 |
| 문서 재업로드 시 기존 그래프 초기화 | PM의 승인 작업이 유실됨 | MVP에서는 초기화 방식으로 시작. 문서를 자주 변경하지 않는 사용 시나리오 타겟 |

## 7. Success Metrics

포트폴리오용 프로젝트로서 비즈니스 KPI보다 기술적 완성도와 데모 품질을 기준으로 측정한다.

| Metric | Target | 측정 방법 | 분모 기준 |
|--------|--------|-----------|-----------|
| 데모 시나리오 성공률 | 3개 문서 → 영향 분석까지 완료 | 샘플 문서 세트로 테스트 | — |
| 파싱 정확도 | PM 검토 시 수정률 30% 이하 | 샘플 문서 기준 수동 평가 | 수정한 요구사항 수 / 총 파싱된 요구사항 수 |
| 의존관계 추론 정확도 | PM 승인률 60% 이상 | 승인/거부 비율 추적 | 승인 수 / (승인 + 거부) 수 |
| End-to-end 소요 시간 | 업로드부터 영향 분석까지 5분 이내 | 타이머 측정 | — |

## 8. v2 Roadmap

MVP 이후 검토할 기능들을 정리한다.

| 기능 | 설명 |
|------|------|
| Diff 기반 부분 업데이트 | 문서 재업로드 시 변경된 부분만 감지하여 그래프를 부분 업데이트. 기존 승인 유지. |
| N-hop 영향 전파 | 직접 연결뿐 아니라 간접 연결까지 영향 범위를 확장. 확산 예상 경로 표시. |
| 템플릿 지원 | 권장 문서 템플릿 제공으로 파싱 정확도 향상. |
| 교차 문서 병합 및 복수 span | 서로 다른 문서의 요구사항 병합, 복수 source span 모델 지원. |
